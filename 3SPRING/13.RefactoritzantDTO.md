# DTO per a Províncies en Spring Boot

Hem de separar **el que entra i ix per l’API** del que és **la nostra entitat JPA**. Açò ens dona una API més estable, més segura i més fàcil de mantindre.

En este exemple refactoritzarem **només Províncies** amb els **mínims canvis possibles**, afegint DTOs **entre el controlador i l’entitat**.

---

## Què aconseguim amb DTOs

* **No exposem l’entitat JPA** directament (evitem acoblaments i problemes de serialització).
* Controlem millor **els camps que acceptem** en POST/PUT.
* Podem afegir **validació** de dades d’entrada sense embrutar l’entitat.
* La resposta del GET pot incloure camps útils (per exemple `paisNom`) sense tocar la BD.

---

## Pas 1. Crear els DTOs

Crearem dos DTOs perquè habitualment no ens interessa que el que entra siga igual que el que ix.

* **Request DTO**: per a POST i PUT (entrada).
* **Response DTO**: per a GET (eixida).

Açò és necessari perquè:

* en l’entrada no volem que ens envien un objecte `Pais` complet, només un `paisId`.
* en l’eixida volem tornar `id`, `nom` i també informació del país (si interessa).

---

### 1.1. Crear `ProvinciaRequest`

A la carpeta DTO del nostre projecte creem el fitxer:
`.../model/dto/ProvinciaRequest.java`

```java
package com.jaume.Provincies.model.dto;

import jakarta.validation.constraints.NotBlank;
import jakarta.validation.constraints.NotNull;

public class ProvinciaRequest {

    @NotBlank(message = "El nom és obligatori")
    private String nom;

    @NotNull(message = "El paisId és obligatori")
    private Long paisId;

    public String getNom() {
        return nom;
    }

    public void setNom(String nom) {
        this.nom = nom;
    }

    public Long getPaisId() {
        return paisId;
    }

    public void setPaisId(Long paisId) {
        this.paisId = paisId;
    }
}
```

**Este pas és necessari perquè:**

* Evitem que el client envie coses com:

  * `id` (no l’ha de decidir el client en un POST)
  * un objecte `pais` complet
* Fem que el JSON d’entrada siga simple i clar:

  * `nom`
  * `paisId`
* Podem validar abans de guardar:

  * si `nom` ve buit, falla
  * si no ve `paisId`, falla

---

### 1.2. Crear `ProvinciaResponse`

Creem el fitxer a `../model/dto/ProvinciaResponse.java`

```java
package com.jaume.Provincies.model.dto;

public class ProvinciaResponse {

    private Long id;
    private String nom;
    private Long paisId;
    private String paisNom;

    public ProvinciaResponse() {}

    public ProvinciaResponse(Long id, String nom, Long paisId, String paisNom) {
        this.id = id;
        this.nom = nom;
        this.paisId = paisId;
        this.paisNom = paisNom;
    }

    public Long getId() {
        return id;
    }

    public void setId(Long id) {
        this.id = id;
    }

    public String getNom() {
        return nom;
    }

    public void setNom(String nom) {
        this.nom = nom;
    }

    public Long getPaisId() {
        return paisId;
    }

    public void setPaisId(Long paisId) {
        this.paisId = paisId;
    }

    public String getPaisNom() {
        return paisNom;
    }

    public void setPaisNom(String paisNom) {
        this.paisNom = paisNom;
    }
}
```

**Este DTO fa que:**

* La resposta del GET no depén de l’entitat JPA.
* Podem tornar exactament els camps que ens interessen.
* Podem incloure `paisNom` per fer la resposta més útil sense tocar BD ni entitats.
* Evitem retornar relacions completes (ciutats, etc.) si no toca.

---

## Pas 2. Afegir el mapatge (entitat → DTO)

Ara hem de convertir una `Provincia` (entitat) a un `ProvinciaResponse`.

Este pas és necessari perquè:

* el controlador treballa amb entitats (el repositori retorna entitats)
* però l’API ha de retornar DTOs

---

### 2.1. Afegir imports al controlador

En el fitxer del controlador (on tens els endpoints), afegix:

```java
import com.jaume.Provincies.model.dto.ProvinciaRequest;
import com.jaume.Provincies.model.dto.ProvinciaResponse;
import jakarta.validation.Valid;
import org.springframework.web.bind.annotation.RequestBody;
```

**Amb estos imports aconseguim:**

* `ProvinciaRequest` i `ProvinciaResponse` s’utilitzen en les signatures dels endpoints.
* `@Valid` fa que la validació del Request s’execute automàticament.
* `@RequestBody` permet rebre JSON en el cos de la petició.

---

### 2.2. Crear el mètode `toProvinciaResponse` al controlador

Dins del controlador, afegix este mètode privat:

```java
private ProvinciaResponse toProvinciaResponse(Provincia provincia) {
    if (provincia == null) return null;

    Long paisId = null;
    String paisNom = null;

    if (provincia.getPais() != null) {
        paisId = provincia.getPais().getId();
        paisNom = provincia.getPais().getNom();
    }

    ProvinciaResponse response = new ProvinciaResponse();
    response.setId(provincia.getId());
    response.setNom(provincia.getNom());
    response.setPaisId(paisId);
    response.setPaisNom(paisNom);

    return response;
}
```

**Amb este mètode:**

* Evitem repetir codi en cada endpoint.
* Centralitzem el format de la resposta en un sol lloc.
* Si demà canviem el DTO, només toquem este mètode.
* Ens protegim davant `NullPointerException` si `pais` fora nul.

---

## Pas 3. Canviar els endpoints de Províncies

Ara canviarem el controlador perquè:

* **reba** `ProvinciaRequest` en POST i PUT
* **torne** `ProvinciaResponse` en GET, POST i PUT

Açò és necessari perquè:

* el client no parla amb entitats
* el client parla amb DTOs

---

### 3.1. GET totes les províncies

```java
@GetMapping("/provincies")
public List<ProvinciaResponse> getProvincies() {
    List<Provincia> provincies = provinciaRepository.findAll();
    List<ProvinciaResponse> resposta = new ArrayList<>();

    for (Provincia provincia : provincies) {
        resposta.add(toProvinciaResponse(provincia));
    }

    return resposta;
}
```

**Per què és necessari este canvi?**

* `findAll()` retorna entitats.
* nosaltres volem retornar DTOs.
* mapegem cada entitat a `ProvinciaResponse`.

---

### 3.2. GET província per id

```java
@GetMapping("/provincies/{id}")
public ProvinciaResponse getProvincia(@PathVariable Long id) {
    Provincia provincia = provinciaRepository.findById(id).orElse(null);
    return toProvinciaResponse(provincia);
}
```

**Per què és necessari este canvi?**

* Tornem un DTO consistent amb el GET de llista.
* No exposem l’entitat ni les relacions.

---

### 3.3. POST crear província

```java
@PostMapping("/provincies")
public ProvinciaResponse createProvincia(@Valid @RequestBody ProvinciaRequest request) {
    Pais pais = paisRepository.findById(request.getPaisId()).orElse(null);
    if (pais == null) return null;

    Provincia provincia = new Provincia();
    provincia.setNom(request.getNom());
    provincia.setPais(pais);

    Provincia guardada = provinciaRepository.save(provincia);
    return toProvinciaResponse(guardada);
}
```

**Per què és necessari este canvi?**

* El client envia `paisId`, nosaltres busquem l’entitat `Pais`.
* Evitem que el client ens envie un `Pais` manipulat o incomplet.
* Guardem l’entitat i tornem un DTO (resposta neta).

---

### 3.4. PUT actualitzar província

```java
@PutMapping("/provincies/{id}")
public ProvinciaResponse updateProvincia(@PathVariable Long id,
                                        @Valid @RequestBody ProvinciaRequest request) {

    Provincia existent = provinciaRepository.findById(id).orElse(null);
    if (existent == null) return null;

    Pais pais = paisRepository.findById(request.getPaisId()).orElse(null);
    if (pais == null) return null;

    existent.setNom(request.getNom());
    existent.setPais(pais);

    Provincia guardada = provinciaRepository.save(existent);
    return toProvinciaResponse(guardada);
}
```

**Per què és necessari este canvi?**

* L’`id` ve de la URL (això és més REST correcte).
* Recuperem primer l’entitat per evitar sobreescriure camps no modificats.
* Les dades editables venen del DTO (`nom` i `paisId`).
* Tornem un DTO consistent amb la resta d’endpoints.

---

### 3.5. DELETE

Es queda igual:

```java
@DeleteMapping("/provincies/{id}")
public void deleteProvincia(@PathVariable Long id) {
    provinciaRepository.deleteById(id);
}
```

**Per què no fem DTO ací?**

* No hi ha cos (`RequestBody`).
* No hi ha un objecte a retornar.
* Un DELETE normal només necessita l’`id`.

---

# Refactoritzar tot el projecte amb DTOs

Estos canvis s’han d’aplicar a **tot el projecte** si volem que **cap controlador expose entitats JPA** directament. El procediment és sempre el mateix: **per a cada entitat que ix o entra per l’API, creem DTOs i fem un mapatge**.

La idea és fer-ho **amb els mínims canvis**, com hem fet amb Províncies: **sense Services, sense MapStruct, sense reestructurar mig projecte**.

---

### Criteri general

Hem de decidir, per a cada entitat, dos coses:

* **Què entra** en POST/PUT (Request DTO)
* **Què ix** en GET (Response DTO)

I, quan hi ha relacions (`@ManyToOne`, `@OneToMany`), en els DTOs:

* En l’entrada: **usem IDs** (ex: `provinciaId`, `paisId`)
* En l’eixida: podem tornar **IDs i noms** (ex: `provinciaId`, `provinciaNom`) i evitar retornar l’objecte sencer.

Amb estos canvis aconseguim:

* evitar JSON massa grans o amb relacions circulars
* evitar problemes de serialització amb Lazy
* fer l’API més estable i clara

---

## Procediment (mateixos passos per a totes les entitats)

### Pas 1. Identificar on s’exposa l’entitat

Busquem endpoints que fan:

* `public Entitat ...`
* `public List<Entitat> ...`
* `@RequestBody Entitat ...`

Estos endpoints són els que hem de refactoritzar a DTO.

---

### Pas 2. Crear els DTOs (Request i Response)

En `src/main/java/com/jaume/Provincies/model/dto/`:

* `XxxRequest`
* `XxxResponse`

Amb estos DTOs:

* l’entrada i l’eixida no sempre tenen els mateixos camps
* ens permet decidir quins camps exposem

---

### Pas 3. Afegir validació en els Request DTO

En els `Request`:

* `@NotBlank`, `@NotNull`, etc.

I en el controlador:

* `@Valid`

Necessari perquè:

* validem abans de guardar en BD
* evitem dades incompletes o inconsistents

---

### Pas 4. Crear mapatges en el controlador

Afegim mètodes privats:

* `toXxxResponse(Entitat e)`
* (opcional) `toEntity(XxxRequest dto)` o construcció manual dins del POST/PUT

Amb estos mètodes aconseguim que:

* el repositori treballa amb entitats
* l’API treballa amb DTOs
* evitem repetir el mateix `new Response(...)` en cada endpoint

---

### Pas 5. Canviar signatures dels endpoints

Canvis típics:

* `GET`

  * de `Entitat` → a `XxxResponse`
* `POST` / `PUT`

  * de `@RequestBody Entitat` → a `@Valid @RequestBody XxxRequest`
  * després guardem entitat i retornem `XxxResponse`

Amb estes signatures fem que:

* el client no “parla” amb entitats
* el client envia dades simples (IDs) i rep respostes netes
